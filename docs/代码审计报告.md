## 二、严重问题 (CRITICAL) - 必须修复

### 2.1 敏感凭证硬编码

**风险等级**: 严重

**问题描述**: 配置文件中的数据库密码、API Key、JWT 密钥等敏感信息均为明文硬编码，存在严重泄露风险。

**涉及文件**:
- `server/config/config.yaml`
- `pdf-service/app/core/config.py`

**具体问题**:

```yaml
# server/config/config.yaml
db:
  password: "jianqiang001223"          # 数据库密码明文

jwt:
  secret: "kaoyan-jwt-secret-change-in-production"  # JWT 密钥未修改

dashscope:
  api_key: "sk-api-ULv2Hrbqb-..."     # API Key 泄露

minio:
  secret_key: "Jianqiang001223@"       # MinIO 密钥硬编码
```

**建议修复**:
1. 引入环境变量或 `.env` 文件管理敏感配置
2. 生产环境使用密钥管理服务 (如 HashiCorp Vault)
3. 启动时检查是否为默认值，若否则拒绝启动

---

### 2.2 内部 API 缺乏认证保护

**风险等级**: 严重

**问题描述**: 内部回调接口 `/api/v1/internal/*` 完全没有认证保护，任意用户均可调用。

**涉及文件**: `server/internal/router/router.go` 第 60-64 行

```go
// 内部回调 (无需用户认证，应通过内部网络或API Key保护)
internal := v1.Group("/internal")
{
    internal.POST("/textbooks/parse-callback", r.pdfHandler.ParseCallback)
}
```

**建议修复**:
1. 添加 API Key 验证中间件
2. 使用内部网络隔离
3. 验证请求来源 IP

---

### 2.3 MongoDB 认证凭证泄露

**风险等级**: 严重

**问题描述**: MongoDB 连接字符串中包含明文用户名和密码。

```yaml
mongodb:
  uri: "mongodb://kaoyan:kaoyan_mongo_2026@localhost:27017/..."
```

**建议修复**: 使用环境变量管理

---

## 三、高优先级问题 (HIGH)

### 3.1 微信登录开发模式静默返回 Mock 数据

**涉及文件**: `server/internal/pkg/wechat/client.go` 第 39-46 行

```go
func (c *Client) Code2Session(code string) (*Code2SessionResponse, error) {
    if c.appID == "" || c.appSecret == "" {
        // 开发模式：返回模拟数据
        return &Code2SessionResponse{
            OpenID:     "mock_openid_" + code,
            SessionKey: "mock_session_key",
        }, nil
    }
    // ...
}
```

**问题**: 生产环境若未配置微信密钥，系统会静默使用 mock 数据，导致用户数据混乱。

**建议**:
1. 明确区分开发/生产模式
2. 生产环境缺少配置时应启动失败
3. 添加启动警告提示

---

### 3.2 定时任务无并发控制

**涉及文件**: `server/main.go` 第 98-127 行

**问题**: 每日凌晨刷新所有活跃计划时串行执行，用户量大的情况下耗时过长。

```go
func startDailyCron(plannerService *service.PlannerService, logger *zap.Logger) {
    for {
        // ...
        for _, plan := range plans {
            // 串行执行刷新
            if _, err := plannerService.RefreshPlan(plan.ID); err != nil {
                // ...
            }
        }
    }
}
```

**建议**:
1. 使用 Go 协程池并发处理
2. 添加超时机制
3. 考虑分布式锁防止重复执行

---

### 3.3 AI 调用超时设置过长

**涉及文件**: `server/internal/pkg/dashscope/client.go` 第 24-26 行

```go
client: &http.Client{
    Timeout: 300 * time.Second,  // 5 分钟超时
}
```

**问题**: 5 分钟超时过长，用户体验差且浪费资源。

**建议**: 根据 AI 模型设置合理超时 (30-60 秒)

---

## 四、中等问题 (MEDIUM)

### 4.1 PDF 服务 CORS 配置过于宽松

**涉及文件**: `pdf-service/main.py` 第 17-24 行

```python
app.add_middleware(
    CORSMiddleware,
    allow_origins=["*"],  # 允许所有来源
    allow_credentials=True,
    allow_methods=["*"],
    allow_headers=["*"],
)
```

**建议**:
1. 指定具体允许的域名
2. 生产环境不建议同时使用 `allow_credentials=True` 和 `allow_origins=["*"]`

---

### 4.2 任务状态更新无事务保护

**涉及文件**: `server/internal/service/planner.go` 第 289-318 行

**问题**: 更新任务状态和计划统计是两个独立操作，若后者失败会导致数据不一致。

```go
func (s *PlannerService) UpdateTaskStatus(taskID uint64, status string) error {
    // 1. 更新任务状态
    if err := s.taskRepo.Update(task); err != nil {
        return err
    }

    // 2. 更新计划统计 (可能失败)
    total, completed, _ := s.taskRepo.GetCompletionStats(task.PlanID)
    // ...
}
```

**建议**: 使用数据库事务包裹两个操作

---

### 4.3 日志可能泄露用户信息

**涉及文件**: 多处

```go
logger.Info("刷新计划", zap.Uint64("plan_id", plan.ID), zap.String("plan_name", plan.PlanName))
```

**建议**: 避免记录敏感信息，添加敏感字段过滤

## 五、性能优化点 (PERFORMANCE)

### 5.1 聊天历史无限制

**涉及文件**: `server/internal/service/chat.go` 第 73-75 行

```go
var history []model.ChatMessage
s.db.Where("session_id = ?", sessionID).Order("created_at ASC").Find(&history)
// 所有历史消息都用于构建上下文
```

**问题**: 长时间对话会导致 Token 费用激增。

**建议**:
1. 限制最近 N 条消息 (如 20 条)
2. 限制时间范围 (如最近 7 天)
3. 添加消息数量上限

---

### 5.2 数据库连接池配置

**涉及文件**: `server/main.go`

```go
sqlDB.SetMaxOpenConns(cfg.DB.MaxOpenConns)  // 默认 100
sqlDB.SetMaxIdleConns(cfg.DB.MaxIdleConns)  // 默认 10
```

**建议**: 根据实际 QPS 调整，建议添加连接池监控

---

### 5.3 AI 响应解析脆弱

**涉及文件**: `server/internal/service/planner.go` 第 485-500 行

**问题**: AI 返回格式不稳定时会频繁降级到规则引擎。

**建议**:
1. 使用更严格的 Prompt
2. 添加 JSON Schema 验证
3. 考虑使用 Function Calling
