### S2. 多个接口缺少用户归属校验（越权访问）

**涉及接口**:

| 接口 | 文件 | 问题 |
|------|------|------|
| `GET /chat/sessions/:id` | `handler/chat.go:44` | 任意用户可读取他人对话历史 |
| `POST /chat/sessions/:id/messages` | `handler/chat.go:73` | 任意用户可向他人会话发消息 |
| `GET /plans/:id` | `handler/plan.go:42` | 任意用户可查看他人学习计划 |
| `POST /plans/:id/refresh` | `handler/plan.go:132` | 任意用户可刷新他人计划 |
| `PATCH /tasks/:id/status` | `handler/plan.go:107` | 任意用户可修改他人任务状态 |
| `POST /notifications/generate` | `handler/notification.go:45` | 任意用户可为他人计划生成通知 |

**风险**: 只要知道 ID，已认证用户可以操作任何人的数据（IDOR 漏洞）。

**修复方案**: 每个接口在操作资源前，必须校验 `resource.UserID == middleware.GetUserID(c)`。

### S3. 数据库 Schema 与 GORM Model 不一致

**文件**: `database/postgres/init.sql` vs `model/study_plan.go`

`study_plans` 表的 init.sql 中 **缺少以下字段**，但 GORM Model 中存在：

**修复方案**:

2. 或在 `main.go` 中增加 `db.AutoMigrate()` 自动迁移

### S5. 定时任务并发竞态条件

**文件**: `main.go` 第 132-166 行

```go
successCount := 0  // 非线程安全
failCount := 0     // 非线程安全

// 多个 goroutine 并发修改这两个变量
go func(...) {
    successCount++  // 数据竞争！
    failCount++     // 数据竞争！
}(...)
```

**后果**: `successCount` 和 `failCount` 结果不准确，可能丢失计数。

**修复方案**: 使用 `atomic.AddInt64` 或 `sync.Mutex` 保护计数器。

### S6. 定时任务 Goroutine 泄漏

**文件**: `main.go` 第 145-161 行

```go
go func() {
    defer close(done)
    // 刷新逻辑（可能很慢，包含 AI API 调用）
}()

select {
case <-done:
case <-time.After(60 * time.Second): // 超时后...
    // ⚠ 内部 goroutine 仍在运行！无法取消！
}
```

**后果**: 超时后，内部 goroutine 继续运行（包含 AI API 调用），永远不会被回收，造成 goroutine 泄漏。

**修复方案**: 使用 `context.WithTimeout` 传递给刷新方法，超时后取消。

### S7. CORS 配置不区分环境

**文件**: `middleware/middleware.go` 第 14 行


**风险**: 生产环境也会允许任何域名跨域请求，存在 CSRF/数据窃取风险。

**修复方案**: 从配置读取允许的域名列表，开发环境用 `*`，生产环境用具体域名。

### I4. pdf-service CORS 配置违反规范
